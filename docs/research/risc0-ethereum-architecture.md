**Source repository:** `risc0/risc0-ethereum` (sibling directory `../risc0-ethereum/`)

**Relevance:** Cross-validation of our serialization specs; architectural patterns for on-chain

## 2. Three-layer generation model

The on-chain code decomposes into three clearly separated layers:

| Layer | File | How produced |
|-------|------|--------------|
| Raw pairing check + hardcoded VK | `Groth16Verifier.sol` | snarkJS, per trusted setup |
| Version constants (`CONTROL_ROOT`, `BN254_CONTROL_ID`) | `ControlID.sol` | `cargo xtask bootstrap-groth16`, per risc0 release |
| Claim decoding, selector check, public-input wiring | `RiscZeroGroth16Verifier.sol` | Static, version-independent |

**Mapping to ADR-0004:** This confirms our design. gnark produces the BLS12-381 outer
verifier (≈ `Groth16Verifier.sol`). The plugin codegen emits system-specific Aiken logic with
baked-in constants (≈ `ControlID.sol` + `RiscZeroGroth16Verifier.sol` merged). The
version-independent BLS12-381 verification logic stays generic.

## 3. Selector mechanism

`receipt.inner.groth16().seal` from the RISC Zero Rust SDK is **256 bytes with no prefix**.
The 4-byte selector (`0xc27d1bc0` for V5.0) is added by `risc0-ethereum`'s `encode_seal()`
for Ethereum compatibility only — it is not part of our canonical `proof.bin`.

The selector is derived as:
```
SHA-256(
  "risc0.Groth16ReceiptVerifierParameters"
  || control_root
  || reverseByteOrder(bn254_control_id)
  || verifier_key_digest()
)[0:4]
```

`verifier_key_digest()` is a tagged SHA-256 over all VK curve points:
```
SHA-256(
  "risc0_groth16.VerifyingKey"
  || SHA-256(alpha)
  || SHA-256(beta)
  || SHA-256(gamma)
  || SHA-256(delta)
  || taggedList("risc0_groth16.VerifyingKey.IC", [SHA-256(IC[i]) for each i])
)
```

**Selector history** (useful for version compatibility checks):

| Version | Selector |
|---------|----------|
| V5.0 (current) | `0xc27d1bc0` |
| V3.0 | `0x73c457ba` |
| V2.2 | `0xbb001d44` |
| V1.2 | `0xc101b42b` |

## 5. Claim hash chain — independent confirmation

The Solidity `IRiscZeroVerifier.digest()` independently confirms the hash chain documented in
`docs/research/risc0-artifact-format.md §6a`. Key encoding detail confirmed:

```solidity
uint32(claim.exitCode.system) << 24  // high-byte encoding = LE uint32 for values 0-255
```

For `Halted=0, user=0` (normal proofs), all exit code bytes are zero — no difference in
practice, but the bit pattern confirms our spec.

## 6. `verify()` vs `verifyIntegrity()` — two-API pattern

```solidity
// High-level: reconstructs ReceiptClaim internally
verify(seal, imageId, sha256(journal))

// Low-level: takes pre-computed claimDigest
verifyIntegrity(Receipt { seal, claimDigest })
```

The simple form is what applications use. `verifyIntegrity()` is for advanced cases (set
verifiers, aggregation). The verifier never sees raw journal bytes — only `sha256(journal)`.

**Aiken parallel:** Our generated Aiken validator is the equivalent of the high-level form: it
takes raw journal bytes, hashes them on-chain, and drives the full hash chain. A low-level
entry point taking a pre-computed claim digest could be a Phase 3 addition for batching.

---

## 7. Image ID is cryptographically bound — no separate assertion

`verify(seal, imageId, journalDigest)` embeds `imageId` into the ReceiptClaim digest, which
feeds into `inputs[2,3]`. The Groth16 circuit commits to that digest. There is no separate
"is this the right program?" guard — the commitment is cryptographic.

**Confirmed:** baking `imageId` as a constant into the generated Aiken module and asserting
`inputs[2,3] == split_digest(claim_digest(imageId, journalDigest))` is the correct approach.

## 8. Verifier router (governance pattern)

`RiscZeroVerifierRouter` maps 4-byte selectors → `IRiscZeroVerifier` implementations
(`Ownable2Step`). Verifiers are added/removed by the owner (a `TimelockController` in
production). Removed selectors are permanently tombstoned.

Each verifier version has a `RiscZeroVerifierEmergencyStop` wrapper whose guardian can freeze
it independently of the router.

**Cardano equivalent (not yet designed):** Parameterize the Aiken validator with a governance
NFT or use a reference input for a "circuit active" datum. Emergency freeze = burn the NFT or
update the datum. Needs an ADR before deployment design.

## 9. End-to-end user workflow

### Off-chain (proving)

1. Developer writes a **guest program** (Rust → RISC-V ELF). `IMAGE_ID` is derived from the
   ELF at compile time and is fixed.
2. Host program proves with `ProverOpts::groth16()` → `Receipt` containing `seal` (256 bytes)
   and `journal` (guest output bytes).
3. `encode_seal(&receipt)` prepends the 4-byte version selector → 260-byte blob for Ethereum.
4. Host submits a transaction carrying `seal` + raw `journal` bytes.

### On-chain (verification)

The application contract stores `IMAGE_ID` as an immutable constant baked in at deployment
and holds a reference to the shared `IRiscZeroVerifier` router. A typical app contract:

```solidity
contract MyApp {
    IRiscZeroVerifier immutable verifier;
    bytes32 constant IMAGE_ID = 0x5135...;

    function submitResult(bytes calldata seal, bytes calldata journal) external {
        verifier.verify(seal, IMAGE_ID, sha256(journal));
        // journal bytes are now trusted — decode and act on them
        uint256 result = abi.decode(journal, (uint256));
    }
}
```

The verifier call does everything: selector check, `ReceiptClaim` reconstruction from
`IMAGE_ID` + `sha256(journal)`, Groth16 proof verification. The app never touches proof
internals.

### Mapping to Cardano

| Ethereum pattern | Cardano equivalent |
|---|---|
| `IMAGE_ID` hardcoded in app contract | `IMAGE_ID` baked into generated Aiken module at codegen time |
| `sha256(journal)` computed in app contract | Aiken validator hashes raw journal bytes from redeemer on-chain |
| Shared singleton verifier router | One deployed outer verifier script; per-system Aiken modules reference it |
| Proof consumed statelessly in one tx | Same — no proof state persists on-chain |
| Raw journal bytes travel with the tx | Journal bytes in the redeemer; validator drives the full hash chain |

The overall shape: **off-chain prover produces (seal, journal) → on-chain app verifies
(seal, IMAGE_ID, sha256(journal)) → app acts on journal bytes.** Clean, minimal interface.

### Between encode_seal() and transaction submission

`encode_seal()` produces a 260-byte `Vec<u8>` and stops there — the library provides no
wallet, signing, or broadcast utilities. The blob is transport-agnostic (file, HTTP, calldata).
Users wire the submission side themselves with alloy; the only in-repo example is a ~5-line
integration test in `crates/test-utils`. For a full app skeleton, RISC Zero points to the
external `risc0-foundry-template` repo.

The `risc0-forge-ffi` crate provides a CLI that writes `hex(abi.encode(journal, seal))` to
stdout for consumption by Foundry's FFI cheatcode — useful for Solidity test suites, not for
production submission.

## 10. `RiscZeroSetVerifier` — Merkle batching pattern

For scaling, RISC Zero deploys a **per-guest-program** `RiscZeroSetVerifier`. A "set builder"
guest runs off-chain, accumulates many claim digests into a Merkle tree, and produces one
Groth16 proof over the root. On-chain: `verify(proof_of_root)` + `verifyMember(leaf, path)`.

**Cardano parallel (future):** Outer proof certifies a Merkle root; Aiken validator checks
Merkle inclusion. Relevant if per-transaction proof verification hits ExUnits limits or if
batching becomes needed. Note for Phase 3+ planning.
